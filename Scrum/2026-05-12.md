# 3차 회의 : 요구사항

## 사용자 타겟

주 사용자:

- 자신의 기분이나 상황에 맞는 음악을 빠르게 추천받고 싶은 사용자
- 추천받은 곡을 플레이리스트로 저장하고 싶은 사용자
- 복잡한 음악 검색보다 간단한 입력으로 음악을 추천받고 싶은 사용자
## 요구사항 정리

### 기능적 요구사항 

```python
FR-01. 사용자는 로그인할 수 있어야 한다.
FR-02. 사용자는 회원가입할 수 있어야 한다.
FR-03. 사용자는 현재 기분을 선택할 수 있어야 한다.
FR-04. 사용자는 원하는 분위기나 상황을 텍스트로 입력할 수 있어야 한다.
FR-05. 시스템은 기분과 상황 입력값을 바탕으로 음악 추천 결과를 제공해야 한다.
FR-06. 시스템은 추천 결과에 장르 태그를 함께 표시해야 한다.
FR-07. 사용자는 추천받은 곡 목록을 플레이리스트로 저장할 수 있어야 한다.
FR-08. 사용자는 저장한 플레이리스트 목록을 조회할 수 있어야 한다.
FR-09. 사용자는 플레이리스트 상세 정보를 조회할 수 있어야 한다.
FR-10. 사용자는 플레이리스트 제목과 설명을 수정할 수 있어야 한다.
FR-11. 사용자는 플레이리스트를 삭제할 수 있어야 한다.
FR-12. 사용자는 플레이리스트에 곡을 추가할 수 있어야 한다.
FR-13. 사용자는 플레이리스트에서 곡을 삭제할 수 있어야 한다.
FR-14. 사용자는 플레이리스트 내 곡 순서를 변경할 수 있어야 한다.
FR-15. 사용자는 추천 결과에 대해 피드백을 남길 수 있어야 한다.
FR-16. 사용자는 로그아웃할 수 있어야 한다.
```

### 비기능 요구사항

```python
NFR-01. UI는 초보자도 쉽게 사용할 수 있어야 한다.
NFR-02. 추천 요청 후 사용자는 로딩 상태를 확인할 수 있어야 한다.
NFR-03. 저장, 삭제, 수정 후 즉각적인 피드백 메시지를 제공해야 한다.
NFR-04. 삭제 기능은 실수를 방지하기 위해 확인 모달을 제공해야 한다.
NFR-05. 화면은 모바일 환경에서도 사용 가능해야 한다.
NFR-06. 기능별 컴포넌트는 유지보수하기 쉽게 분리되어야 한다.
NFR-07. 실제 API가 없어도 mock data 기반으로 동작해야 한다.
NFR-08. 주요 기능은 테스트 케이스로 검증 가능해야 한다.
```

## 회고 (Today I Learned)

성현 : 

소공 시간에 배운 '요구 분석' 단계를 우리 음악 추천 프로젝트에 직접 적용해봤다. 이전 프로젝트들에서는 대충 이런 기능 만들자고 정한 뒤 코딩부터 시작했는데, 교안에서 배운 대로 요구사항을 기능 요구와 비기능 요구로 명확히 분류해 보니 기획의 밀도가 확 달라졌다. 사용자가 기분을 텍스트로 입력하거나 플레이리스트를 저장하는 기능 요구는 동사로 표현돼서 꽤 쉽게 파악할 수 있었지만 , '초보자도 쉽게 사용할 수 있는 UI' 같은 비기능 요구는 형용사로 표현되다 보니 객관적으로 파악하기 어렵다는 책의 내용이 정말 와닿았다. 


은서 : 

오늘 회의를 통해 소프트웨어 개발의 실질적인 첫 단계인 요구 분석이 실제 아키텍처 설계에 얼마나 큰 영향을 미치는지 체감했다. 사용자 스토리를 `<사용자>는 <목표>를 얻기 위해 <행위>를 원한다` 형식에 맞춰 시나리오 기반으로 분석해 보니  백엔드에서 어떤 리소스를 다루는 API를 설계해야 할지 한결 선명해졌다. 요구 분석 내용이 모호함 없이 참여자 모두에게 명확하게 이해될 수 있어야 한다는 검증 파트의 내용이 제일 중요하게 다가왔다. 나중에 실제 구현과 매핑해서 추적 가능성을 잃지 않으려면  오늘 세세하게 쪼갠 FR, NFR 문서가 든든한 기준표가 될 것 같다.
