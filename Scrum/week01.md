# 프로세스 선정

스크럼

짧은 개발 주기(스프린트)를 반복하면서 팀원 간의 지속적인 소통과 피드백을 통해 프로젝트를 진행하는 방식

### **스크럼의 특징**

- 짧은 주기(스프린트) 단위로 개발을 진행하여 빠르게 결과물을 확인할 수 있음

  ⇒ **스크럼 주기는 최소 1주에 두 번. (화요일, 토요일)**

- 팀원 간의 지속적인 소통과 협업을 중요하게 생각함
- 우선순위가 정해진 백로그를 기반으로 작업을 진행함
- 개발 과정 중 요구사항 변경이나 수정 사항을 유연하게 반영할 수 있음
- 반복적인 개발과 피드백을 통해 프로젝트 완성도를 높일 수 있음

### **스크럼을 선택한 이유**

- 팀원이 2명으로 구성되어 있어 의사소통이 빠르고 협업이 용이하다고 판단
- 프로젝트 진행 중 기능 수정이나 아이디어 변경 가능성이 있어, 변화에 유연하게 대응할 수 있는 방식이 필요했다.
- 짧은 단위로 역할을 나누고 결과를 확인하면서 개발하면 작업 진행 상황을 효율적으로 관리할 수 있다고 판단.
- 작은 규모의 팀에서는 문서 중심의 무거운 프로세스보다, 실제 구현과 피드백 중심의 방식이 더 적합하다고 판단하여 스크럼 방식을 선택!

## 매주 회의 템플릿

회의록은 접근성이 좋은 노션에 수기로 작성하고, 과제 깃 레포리토지에 자동 업로드 파이프라인을 수행 할 수 있게 구성 예정

### 백로그

프로젝트에서 해야 할 작업 목록

기능 추가, 수정 사항, 버그 해결 등의 작업들을 우선순위에 따라 정리해둔다.

- **제품 백로그(Product Backlog)**
  프로젝트 전체에서 필요한 작업 목록
- **스프린트 백로그(Sprint Backlog)**
  현재 스프린트에서 수행할 작업 목록

---

### **사용자 스토리(User Story)**

사용자의 입장에서 필요한 기능을 간단하게 작성한 요구사항

예시: “사용자는 로그인 기능을 통해 자신의 계정에 접근할 수 있어야 한다.”

---

### **스프린트 리뷰(Sprint Review)**

스프린트 종료 후 결과물을 확인하는 과정입니다.

완성된 기능을 시연하고 피드백을 받습니다.

---

### **스프린트 회고(Sprint Retrospective)**

스프린트가 끝난 뒤 팀원들이 프로젝트 진행 방식을 되돌아보는 회의입니다.

주요 내용:

- 잘된 점
- 아쉬운 점

---

### 다음 회의 때 해야할 것

주제에 대한 기능들 좀 더 세부화

노션 문서를 깃 레포지토리에 자동 업로드 되게 하는 파이프라인 구축
